CohortIntelligence

OHDSI-native cohort exploration and hypothesis-generation workbench for OMOP CDM studies.

OMOP CDM v5.3 / v5.4 R + Shiny Read-only CDM access ATLAS JSON cohorts v0.1.0

1. Overview

CohortIntelligence fills the gap between standard cohort diagnostics (CohortDiagnostics, ATLAS) and formal chart review. It provides a data-driven exploration layer that helps researchers understand who is in their cohort, detect unusual subgroups, select informative patients for review, and generate candidate research hypotheses — all directly from OMOP CDM structured data.

Not for clinical decision-making All outputs from CohortIntelligence are exploratory and hypothesis-generating. They are not clinical diagnoses and must not be used to make patient-care decisions. OMOP CDM data are structured administrative and clinical records; they do not represent the complete clinical picture.

What it does

Use caseWhat CohortIntelligence provides
Cohort sanity checkVisual overview of who is in your cohort, what their clinical histories look like, and whether any patients look implausible.
Subgroup discoveryUnsupervised clustering of patients by clinical history. Each cluster may represent a clinically distinct phenotype.
Outlier identificationAnomaly scoring flags patients whose clinical histories are statistically unusual relative to the cohort.
Patient selection for reviewPriority-ranked queue for targeted chart review, rather than random sampling.
Hypothesis generationAutomated statistical comparison between clusters to identify candidate research questions.
Clinician meeting prepStructured patient trajectories and cohort signal reports for multidisciplinary discussion.

3. Launching the Dashboard

Demo mode (no database)

library(CohortIntelligence)
launch_cohort_intelligence()   # opens with 50 synthetic patients

All tabs are fully functional in demo mode. Use it to explore the interface before connecting to a live database.

Live OMOP database

devtools::load_all(".")

connection_details <- DatabaseConnector::createConnectionDetails(
  dbms         = "spark",          # or "postgresql", "sql server", etc.
  server       = "your-server/path",
  user         = "token",
  password     = "your-token",
  port         = 443,
  pathToDriver = "/path/to/jdbc.jar"
)

connection <- DatabaseConnector::connect(connection_details)

launch_cohort_intelligence(
  connection   = connection,
  cdm_schema   = "your_cdm_schema",
  vocab_schema = "your_vocab_schema",
  json_path    = system.file("template", "T2DM.json",
                              package = "CohortIntelligence")
)

DatabaseConnector::disconnect(connection)
Tip: diagnose before launching Run check_cohort_json(connector, json_path) before launch_cohort_intelligence() to verify concept coverage and patient count. A few seconds here prevents a confusing 0-patient result.

5. Loading a Cohort

Option A — ATLAS JSON file (recommended)

No write permissions to the CDM required. CohortIntelligence uses CirceR to compile the JSON into a single read-only SQL SELECT and returns cohort members directly.

json_path <- system.file("template", "T2DM.json",
                          package = "CohortIntelligence")
# or your own ATLAS export:
json_path <- "path/to/my_cohort.json"

launch_cohort_intelligence(
  connection   = connection,
  cdm_schema   = "schema",
  vocab_schema = "schema",
  json_path    = json_path
)

Option B — Pre-built cohort table

launch_cohort_intelligence(
  connection           = connection,
  cdm_schema           = "schema",
  vocab_schema         = "schema",
  cohort_schema        = "results_schema",
  cohort_table         = "cohort",
  cohort_definition_id = 1L
)

Option C — Uploaded RDS file

Select Upload RDS file in the sidebar. The RDS must be a named list with slots: cohort, person, condition, drug, procedure, measurement, observation, visit, death.

What happens after loading

1

Cohort extraction

Cohort members and index dates are retrieved from the JSON or cohort table.

2

Domain extraction

Records from all 7 OMOP domains are pulled for every cohort member, batched in groups of 500 patients to avoid query size limits.

3

Feature engineering

Events are counted per patient × domain × time window relative to the index date, producing a feature matrix.

4

ML pipeline

UMAP dimensionality reduction, k-means/hierarchical clustering, and isolation-forest anomaly scoring run on the feature matrix.

5

All tabs populate automatically

Quilt, UMAP, patient ranking, trajectory (top patient), hypothesis results, and demographics all appear without any further clicks.

6. Cohort Overview — Quilt Plot

The quilt is the centrepiece of the dashboard. It renders a patient × time-window × domain heatmap that lets you see the entire cohort in one glance.

How to read the quilt

CONDITION DRUG VISIT
[-12,-6) [-6,0) [0,6) [-12,-6) [-6,0) [0,6) [-12,-6) [-6,0) [0,6)

  • Each row is one patient. Rows are sorted by cluster (default) so similar patients group together.
  • Each column group is one OMOP domain (CONDITION, DRUG, PROCEDURE, etc.).
  • Each column within a group is a time window relative to the cohort index date. [-6,0) = the 6 months before index.
  • Cell colour intensity = number of events that patient had in that domain during that window. White = 0 events. Saturated colour = many events.
  • Default encoding is log1p(count) — logarithmically compressed so a patient with 50 conditions doesn't wash out the scale.

Understanding time windows

[-18,-12)1.5–1 year beforePrior history

[-6,0)6 months before indexImmediate pre-index period
[0,6)0–6 months afterEarly post-index outcomes
[6,12)6–12 months afterLate outcomes / follow-up

Interpreting patterns

What you seeClinical interpretation
Entire row bright across all domains Data-rich patient with long observation history and frequent healthcare contacts. May represent a complex patient or a well-documented one.
Mostly white row Data-sparse patient. Short observation window, infrequent visits, or incomplete data capture. Review before including in downstream analyses.
Bright condition column before index, dimming after Typical chronic-disease pattern: diagnoses accumulate over years, slow down after the defining event is recorded.
Bright drug column only after index Treatment initiation at cohort entry. Expected for first-prescription cohorts (e.g., T2DM newly started on metformin).
Bands of visually similar rows These are the ML clusters — patients grouped by similar clinical histories. Inspect each band to understand what clinical features define it.
Rows that don't match their neighbours Anomalous patients. May represent distinct clinical subtypes, miscoded records, or patients who transferred care mid-observation.
Short dashes in DEATH column Patients with a recorded death event. Single-event domains appear as isolated marks because they have only one occurrence by definition.
Bright MEASUREMENT + bright DRUG, sparse CONDITION Possible coding pattern: labs and prescriptions are captured but diagnoses are under-coded — common in some EHR configurations or specialty settings.
All patients white in one domain That domain may not be populated in your CDM (e.g., OBSERVATION is often sparse), or the concept extraction missed records due to vocabulary mapping gaps.

7. Anomaly Explorer

UMAP scatter plot

UMAP (Uniform Manifold Approximation and Projection) compresses the patient feature matrix — hundreds of domain × time-window counts — into a 2-D map preserving local structure.

  • Nearby points = clinically similar patients. They share similar patterns of diagnoses, medications, procedures, and visits across time windows.
  • Colour = cluster assignment. Each colour is one algorithmically identified subgroup. Points with the same colour were grouped by the k-means or hierarchical clustering algorithm.
  • Isolated points far from all clusters are the most likely outliers — patients whose clinical histories don't fit any of the dominant patterns.
  • Click any point to select that patient and see their quilt row and trajectory.
Interpretation caution UMAP distances are not linearly interpretable. Two points 5 units apart are not necessarily "twice as different" as two points 2.5 units apart. Focus on the topology (which groups exist, which patients are isolated) rather than exact positions or distances.

Anomaly score

The anomaly score (0–1) comes from an isolation forest — a tree-based algorithm that identifies patients whose feature combinations are rare. High scores indicate patients whose clinical history is difficult to describe with the same rules that work for most of the cohort.

ScoreInterpretationSuggested action
0.0 – 0.4Typical patient — similar to the majority of the cohort.Use as controls when interpreting findings from high-score patients.
0.4 – 0.6Moderately unusual — may be on the boundary between subgroups or have atypical but not rare data.Review in context of their cluster assignment.
0.6 – 1.0Highly unusual — rare combination of clinical features.Prioritise for clinical review. Could be a distinct subtype, data error, or genuinely complex patient.

8. Patient Selector

The Patient Selector is a priority-ranked review queue. It answers: "If I can only review N patients, which ones should I look at first?"

Priority score

ComponentDefault weightWhat it measures
Anomaly score50%How unusual the patient's feature combination is (isolation forest).
Cluster noise flag25%Bonus for patients not assigned to any cluster (ID = −1 or 0). These couldn't be classified — often true outliers.
Data sparsity25%Proportion of (domain × window) cells with zero events. Sparse patients may have incomplete records or a qualitatively different care pattern.

Priority tiers

  • High (red) — Top third. Most unusual, sparse, or unclassified. Review first.
  • Medium (amber) — Middle third. Borderline cases or patients on cluster boundaries.
  • Low (green) — Bottom third. Typical, well-classified, data-rich patients. Useful as controls.
Use case: targeted chart review Export the top 10–20 high-priority patients using the CSV download, pull their full medical records, and bring them to a clinician meeting. This is faster and more informative than random sampling.

9. Trajectory Viewer

The Trajectory Viewer shows the complete clinical timeline for a single selected patient — all recorded OMOP events plotted against days from their cohort index date.

Reading the timeline

  • X-axis — Days relative to the index date. The dashed vertical line at day 0 is the index event.
  • Y-axis — One row per clinical concept (e.g., "Type 2 diabetes mellitus", "Metformin 500 mg", "HbA1c").
  • Each dot — One recorded event. Hover for the exact date, value (for lab results), and full concept name.
  • Colour — Domain (red = condition, blue = drug, green = procedure, purple = measurement, teal = visit).

What to look for

PatternPossible meaning
Dense dots before index, sparse afterPatient has a long prior history but limited follow-up — may have transferred care or left the system.
Drug starts immediately after a condition codeTreatment initiation — clinically expected (e.g., metformin after T2DM diagnosis).
Lab dots cluster around drug changesMonitoring behaviour — lab values being checked in response to treatment changes.
Condition codes only before index, none afterPossible historical coding (condition entered once, never re-entered) or condition resolved.
Very long span between first and last eventLong-term follow-up — suitable for longitudinal analyses.
Single event at very negative dayHistorical record from long before cohort entry — may be relevant as chronic condition history.
Structured data only The trajectory is built from structured EHR records only. It does not include free-text clinical notes, imaging reports, waveforms, or out-of-network events. Always confirm findings with full source records before drawing clinical conclusions.

Controls

  • Domains — Toggle which OMOP domains to show.
  • Top N concepts — Limit each domain to its N most frequent concepts to reduce visual clutter.
  • Day range — Zoom into a specific time period around the index date.

10. Hypothesis Panel

The Hypothesis Panel automatically tests whether clinical features differ between clusters. Results appear automatically after cohort load. Each row is a candidate research question, not a confirmed finding.

Column definitions

ColumnDefinition
cluster_a, cluster_bThe two clusters being compared.
domainOMOP domain the feature belongs to.
concept_nameThe condition, drug, procedure, or measurement tested.
window_labelTime window where the difference was detected.
effect_sizeMagnitude of difference. Rank-biserial correlation (continuous) or Cramér's V (binary). Range 0–1; higher = larger difference.
p_value_adjustedBenjamini-Hochberg corrected p-value. Use as a ranking signal, not a definitive test.
directionWhich cluster has more of this feature (higher_in_cluster_a or higher_in_cluster_b).

Statistical method

  • Continuous features — Wilcoxon rank-sum test; effect size = rank-biserial correlation.
  • Binary features — Fisher's exact test; effect size = Cramér's V.
  • Correction — Benjamini-Hochberg false discovery rate across all pairwise tests.
Circular testing — critical caveat These hypotheses are tested on the same data used to define the clusters. This creates circularity: clusters are designed to be different, so you will almost always find differences between them. Treat all findings as leads for external validation, not as independently confirmed results. Never report these p-values as pre-specified statistical tests.

How to use results

  1. Sort by p_value_adjusted ascending to see the strongest signals.
  2. Read each row as: "Patients in Cluster A have more/fewer [concept] in [window] than Cluster B."
  3. Download the CSV and bring it to a clinician meeting for expert validation of face validity.
  4. Use validated findings to design a formal pre-specified analysis in an independent cohort or using CohortDiagnostics.

Value boxes

BoxMeaningCaveat
N patientsTotal cohort size after all inclusion/exclusion criteria.—
Median follow-upMedian days between cohort entry and exit (cohort_start_date to cohort_end_date).Cohort exit definition varies by JSON. May not reflect actual observation period end.
Median age at indexCalculated from year_of_birth and cohort_start_date.OMOP stores birth year (not date) — age has ±1 year uncertainty.
% with death recordedPatients with any record in the death domain.Lower bound. Death capture in EHR data is notoriously incomplete — in-hospital only, or only if linked to vital statistics.

Data density heatmap

Shows how many cohort patients had at least one event per domain per calendar month. Use it to assess data completeness before drawing conclusions.

  • White columns — No data in that month for that domain. Could be a CDM data lag, a site-level gap, or a genuine absence of events.
  • Domain imbalance — If CONDITION is always bright but MEASUREMENT is mostly white, the CDM may have incomplete lab data or labs may be in the OBSERVATION domain instead.
  • Recent drop-off — Very common. CDM data feeds often have a 3–6 month lag. A sharp drop in the most recent months usually indicates a data lag, not a real clinical change.

12. OMOP Domains Reference

Condition
Diagnoses from condition_occurrence. Includes ICD codes mapped to SNOMED standard concepts.
Drug
Prescriptions and dispensing from drug_exposure. Mapped to RxNorm ingredient level.
Procedure
Clinical procedures from procedure_occurrence. CPT, ICD procedure codes mapped to SNOMED.
Measurement
Lab values and vitals from measurement. Includes numeric values (e.g., HbA1c = 7.2).
Observation
Clinical observations from observation. Includes social history, clinical findings, survey results.
Visit
Healthcare encounters from visit_occurrence. Includes inpatient, outpatient, ER.
Death
Mortality events from death table. Single event per patient. Often incomplete — only captures in-system deaths.

13. Cohort Templates

T2DM.json — Type 2 Diabetes Mellitus on Metformin

This template selects adults with a new T2DM diagnosis who initiate metformin therapy within one year.

#CriterionDetail
1T2DM diagnosisSNOMED 443732, 201826, 4193704 — plus all descendants and mapped codes (ICD-10: E11.*)
2Age ≥ 18At the index date (first qualifying T2DM occurrence)
3Prior observation ≥ 365 daysMust have 365 days of continuous observation period before the index date
4Metformin prescriptionRxNorm 1503297 (metformin, all formulations) within 0–365 days after index
5First occurrence onlyOnly the first-ever qualifying T2DM event counts as index
Why this returns 0 patients in some CDMs The 365-day prior observation window is the most common filter. Patients registered less than a year before their first T2DM code fail this gate. The metformin requirement excludes T2DM patients managed with diet, other agents, or insulin first. Run check_cohort_json() to see how many patients clear each gate.

This template selects adult dermatomyositis patients confirmed by two separate diagnoses (a standard phenotyping approach to improve specificity).

14. Recommended Workflow

1

Verify cohort before launching

Run check_cohort_json(connector, json_path) in the R console. Confirm concept IDs are in the vocabulary and candidate_count > 0.

2

Check Demographics first

Confirm cohort size, age distribution, and data density. If the density heatmap shows white columns in recent months, account for data lag in your analysis window.

3

Read the Quilt at a glance

Sort by cluster (default). Do the cluster bands look visually distinct? Are any domains uniformly empty? Any patients that clearly don't belong with their neighbours?

4

Explore the Anomaly Explorer

Identify isolated points on the UMAP. Note whether they have high anomaly scores. These are candidates for clinical review.

5

Review 3–5 high-priority trajectories

From the Patient Selector, click the top high-priority patients. Do their trajectories look like genuine clinical outliers or like data-quality issues (impossible chronology, all events on one day)?

6

Validate hypotheses with a clinician

Download the Hypothesis Panel CSV. For each top finding, ask: "Does this difference between clusters make clinical sense? Is it a known pattern or a new signal worth investigating?"

7

Iterate

Change time window ranges, try binary vs. log1p encoding, filter to a single cluster, or adjust the priority weights. Each view reveals different structure in the cohort.

15. Troubleshooting

ProblemMost likely causeFix
0 patients after JSON load Prior observation window or drug inclusion rule filtering everyone out Run check_cohort_json(connector, json_path) to diagnose each gate; try a simpler JSON with no drug inclusion rule
argument is of length zero CirceR SQL generation returned empty output Re-run with fetch_cohort_from_json(connector, json_path, verbose = TRUE) to inspect the SQL
INSUFFICIENT_PERMISSIONS CDM schema requires write access for temp tables CohortIntelligence uses read-only CTEs — ensure you're using the latest version
UMAP/clustering not running uwot or isotree not installed install.packages(c("uwot", "isotree"))

Very slow loading on large cohort Feature matrix has thousands of concepts Increase min_concept_freq (default 10) or narrow time windows

16. Limitations & Caveats

LimitationImplication for interpretation
Structured data onlyFree-text notes, imaging, waveforms, and out-of-network events are absent. The "clinical picture" in CohortIntelligence is always partial.
Vocabulary mappingIf your site uses unmapped local codes, those events will not appear. Use check_cohort_json() to verify concept coverage before analysing results.
ML is feature-engineering-sensitiveChanging time window breaks, minimum concept frequency, or value encoding can significantly change cluster assignments and anomaly scores.
Death capture incompleteOnly in-system deaths are captured. True mortality rates are almost always higher than what the OMOP death domain shows.
Small cohorts (<30 patients)UMAP layout is unreliable, isolation forest scores are unstable, and hypothesis tests are severely underpowered.
Circular hypothesis testingHypotheses are tested on the same data used to define clusters. All findings require external validation before reporting.
No causal inferenceCohortIntelligence identifies associations and patterns. It cannot establish causation, counterfactuals, or treatment effects.

17. Frequently Asked Questions

My cohort has 546 candidate patients but the dashboard shows 0. Why?

The candidate count from check_cohort_json() counts patients with any matching condition code — before inclusion rules are applied. The most common filters that remove all patients are: (1) the 365-day prior observation window in the T2DM template, and (2) the metformin-within-365-days rule. Run fetch_cohort_from_json(connector, json_path, verbose = TRUE) to print the full SELECT SQL and identify which JOIN or WHERE clause is removing everyone.

Why does the quilt show "Unassigned (0)" for all patients?

This means the clustering step did not assign any patient to a named cluster, usually because: (a) the feature matrix is too sparse (most patients have no events in most windows), (b) all patients ended up in a single cluster, or (c) the ML pipeline failed silently due to a missing package. Check the R console for [CI] messages after loading.

Can I use my own cohort definition?

Yes. Export any cohort from ATLAS (Export → JSON) and pass it via json_path. Any ATLAS cohort JSON valid for OMOP CDM ≥ 5.0 works. CohortIntelligence does not require an ATLAS server — only the JSON file.

Does CohortIntelligence write anything to the CDM?

No. The cohort SQL generated from a JSON file is executed as a single read-only CTE-based SELECT. No tables, temp tables, or views are created in the CDM schema. You need only SELECT permission on the CDM and vocabulary schemas.

What is the minimum cohort size for meaningful results?

At least 30 patients for basic exploration. For UMAP to produce meaningful topology, 50–100+ is recommended. Hypothesis testing between clusters requires at least 10–15 patients per cluster for the tests to have any power.

The Hypothesis Panel says "No significant hypotheses found." What does that mean?

Either: (a) there are genuinely no statistically different features between clusters at the current effect-size threshold, or (b) there is only one cluster. Try lowering Min effect size to 0.1 or re-running with max_hypotheses = 50.

How do I export results for a clinician who doesn't use R?

  • Hypothesis Panel → Download CSV
  • Quilt → Download PNG or SVG
  • Patient Selector → export the DT table (copy-paste or print)
  • In R: export_cohort_results(results, path = "output/")
